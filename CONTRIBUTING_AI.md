# Contributing — AI Agents

Инструкция для любых AI-агентов (Cursor, Claude, Codex, и т.д.), работающих с репозиторием **AI Engineering Delivery Monitor**.

Люди-разработчики тоже могут опираться на эти правила.

---

## 1. Перед любой работой

1. Прочитай [docs/ai_context.md](./docs/ai_context.md).
2. Прочитай [docs/session_log.md](./docs/session_log.md) (текущее состояние за 5 минут).
3. Открой документы по теме задачи (таблица в `ai_context.md`).
4. Не начинай с кода, если стадия — документация / Discovery.

---

## 2. Source of Truth

- Истина — **Markdown в `docs/`** и корневые инженерные файлы.
- Cursor Canvas **не** source of truth.
- Не опирайся на историю чата как на канон, если она противоречит `docs/`.

---

## 3. Архитектура

- Не нарушай принятые ADR (`docs/adr/`, индекс в `docs/decisions.md`).
- Не добавляй технологии (Kafka, Redis, GraphQL, CQRS, микросервисы, …) без:
  - явного запроса человека, **и**
  - нового ADR со статусом Accepted (или Proposed + явное «ждём решения»).
- Не хардкодь платформы. Используй **Workstream Type**.
- Seed типов текущей команды — данные: `backend`, `frontend`, `oracle`, `qa`.

---

## 4. Если меняется архитектура

1. Обнови соответствующие `.md` (см. таблицу в `docs/ai_context.md` / `docs/README.md`).
2. Создай или обнови ADR в `docs/adr/` по шаблону `0000-template.md`.
3. Обнови индекс `docs/decisions.md`.
4. Добавь запись в `docs/changelog.md`.
5. При крупном этапе — запись в `docs/session_log.md`.
6. После крупного этапа реализации — прогони [smoke checklist](./backend/README.md#smoke-checklist-после-крупного-этапа) (старт, Liquibase, health, sync/jira, 401).

**Нет обновления docs = изменение не считается завершённым.**

---

## 5. Код приложения

- Каталоги: `backend/`, `frontend/`, `docker/`, `scripts/`.
- Не пиши код приложения, пока в `session_log.md` / `roadmap.md` не зафиксирован старт этапа реализации (Skeleton+), либо пока человек явно не попросил код.
- Не коммить секреты. Используй примеры `*.env.example`, не реальные токены.
- Не используй `C:\Program Files\...` как корень проекта.

---

## 6. Стиль документации

- Документы готовы к Git as-is.
- Один крупный документ — один файл (или ADR на решение).
- Термины — по [docs/glossary.md](./docs/glossary.md).
- Язык документации проекта: русский (допустимы английские идентификаторы кода/API).

---

## 7. PR / коммиты (когда код появится)

- Следуй `.github/pull_request_template.md`.
- В описании: зачем, что изменилось в docs/ADR, как проверить.
- Не force-push в `main` без явной просьбы человека.

---

## 8. Запрещено

| Запрет | Почему |
|---|---|
| Ломать ADR «тихо» | Теряется история решений |
| Canvas вместо docs | Не переживает Git/онбординг |
| Код раньше утверждённой архитектуры этапа | Техдолг и хаос |
| Новые зависимости «на всякий случай» | Over-engineering |
| Хардкод Backend/Android/… | Нарушает ADR-002 |
