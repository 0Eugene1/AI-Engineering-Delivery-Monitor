## Security Assumptions

На текущем этапе проекта предполагается:

- приложение работает **только во внутренней сети** компании;
- доступ осуществляется через **корпоративный VPN**;
- сервис **не публикуется в Интернет**;
- все интеграции выполняются через **сервисные аккаунты**.

Эти допущения — базовый контекст для принципов ниже (особенно §4 Service Accounts и §6 Network). При изменении любого из них модель угроз (§1) и сетевые требования (§6) пересматриваются, а изменение фиксируется по [Change policy](#change-policy).

## 1. Security Goals

### Что защищаем

Monitor — **внутренний read-only dashboard** ([architecture.md](./architecture.md)): собственных бизнес-данных он не порождает, а проецирует данные Jira / GitLab / Jenkins. Поэтому в фокусе безопасности:


| Актив                                                                   | Почему важен                                                                                             |
| ----------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| **Service-account credentials** (Jira PAT, GitLab token, Jenkins token) | Дают чтение корпоративных систем разработки; утечка = доступ к Jira/GitLab/Jenkins от имени сервиса      |
| **Спроецированные данные** (issues, коммиты, MR, builds, activity feed) | Внутренняя информация о разработке и релизах; не для внешнего доступа                                    |
| **Доступ к самому Monitor**                                             | Кто видит dashboard и кто может запускать привилегированные действия (напр. `POST /api/admin/sync/jira`) |
| **Целостность логов и аудита**                                          | Основа для расследования инцидентов                                                                      |




### Модель угроз


| Угроза                                 | Вектор                                        | Митигация (принцип)                                     |
| -------------------------------------- | --------------------------------------------- | ------------------------------------------------------- |
| Утечка токенов интеграций              | Секрет в Git, в логах, в конфиге              | Только env vars; запрет секретов в Git и логах (§5, §8) |
| Несанкционированный доступ к dashboard | Отсутствие аутентификации / открытый порт     | Корпоративный SSO + внутренняя сеть/VPN (§2, §6)        |
| Превышение прав пользователем          | Плоская модель доступа                        | Роли + least privilege (§3)                             |
| Компрометация сервисного аккаунта      | Избыточные права токена                       | Минимально необходимые (read-only) права (§4)           |
| Перехват трафика                       | HTTP вместо HTTPS, доступ снаружи             | HTTPS + внутренний сервис за VPN (§6)                   |
| Скрытые злоупотребления                | Нет журналирования привилегированных действий | Audit (§7)                                              |


**Вне модели угроз (осознанно):** Monitor не является точкой атаки на прод-системы — он их только читает; защита самих Jira/GitLab/Jenkins — ответственность их администраторов. Публичный интернет-доступ к Monitor **не предполагается** (§6).

## 2. Authentication

- **Целевая модель — корпоративный провайдер (SSO / LDAP / OIDC).** Аутентификация пользователей в итоге — только через него; конкретный протокол выбирается при развёртывании под инфраструктуру `eltc.ru`.
- **Локальных пользователей в Monitor нет.** Приложение не хранит логины/пароли, не имеет собственной регистрации и восстановления пароля.
- Учётные записи, роли и деактивация сотрудников управляются на стороне корпоративного каталога — Monitor доверяет ему как источнику идентичности.
- Отсутствие своей базы паролей снимает целый класс угроз (утечка хешей, слабые пароли, self-service reset).

### Текущий baseline (до корпоративного SSO/OIDC) — [ADR-012](./adr/0012-minimal-auth-baseline-admin-endpoints.md)

Phase 2.4 вводит первый привилегированный HTTP-endpoint (`POST /api/admin/sync/jira`), поэтому опоры только на сеть/VPN (§6) недостаточно. Принят **минимальный auth-baseline**:

- **Spring Security добавляется только как минимальный enforcement-слой** — защита привилегированных endpoint, проверка admin-доступа и подготовка основы под будущий OIDC. Это **не** полная система аутентификации пользователей.
- **Auth baseline — статичный Bearer API-token из environment variable для `/api/admin/**`.** Ограничения baseline: нет локальных пользователей, нет user storage, нет ролей PM/QA/Developer в коде (только admin-gate), нет OIDC implementation.
- API-token — **временный machine/admin baseline** до выбора корпоративного SSO/OIDC, а **не** финальная аутентификация пользователей. Роли PM/QA/Developer (§3) и защита read-эндпоинтов включаются вместе с SSO и появлением UI.

> Реализация (какой именно провайдер, библиотека, поток OIDC) не входит в этот документ. Введение Spring Security ограничено enforcement-слоем выше и зафиксировано в [ADR-012](./adr/0012-minimal-auth-baseline-admin-endpoints.md).

## 3. Authorization

Модель ролей (минимальный набор для MVP):


| Роль          | Назначение           | Доступ                                                                                    |
| ------------- | -------------------- | ----------------------------------------------------------------------------------------- |
| **Admin**     | Эксплуатация Monitor | Всё чтение + привилегированные действия (напр. `POST /api/admin/sync/jira`), конфигурация |
| **PM**        | Менеджмент поставки  | Чтение всех дашбордов: Board, Timeline, Activity Feed, Release Health, Risks              |
| **QA**        | Контроль качества    | Чтение дашбордов (тот же read-scope, что PM)                                              |
| **Developer** | Разработчик          | Чтение дашбордов                                                                          |




### Принцип минимальных прав (least privilege)

- По умолчанию — **только чтение**. Привилегированные (admin) операции доступны **только** роли Admin.
- Роль не даёт больше, чем нужно для задачи; расширение прав — явное решение, а не побочный эффект.
- Read-only природа продукта ([architecture.md](./architecture.md)) означает, что для большинства ролей запись в принципе отсутствует — писать пользовательские данные в Monitor нечего.
- Набор ролей конфигурируем и может уточняться; это **не** доменная модель и не требует нового ADR при изменении списка.

## 4. Service Accounts

Monitor обращается к внешним системам через **отдельные сервисные аккаунты**, а не через личные учётки сотрудников ([discovery.md](./discovery.md) §7). Каждый аккаунт получает **минимально необходимые (read-only) права**.


| Система     | Что нужно читать                                       | Минимальные права                                    | Чего быть НЕ должно                                           |
| ----------- | ------------------------------------------------------ | ---------------------------------------------------- | ------------------------------------------------------------- |
| **Jira**    | Issues, changelog, comments, links, board/filter 30532 | Read (просмотр проекта `MPTPSUPP` / нужных фильтров) | Права на редактирование issue, администрирование, transitions |
| **GitLab**  | Push/branch, commits, MR, notes/approvals              | `read_api`, `read_repository` (project/group token)  | `write_repository`, `api` с записью, admin scope              |
| **Jenkins** | Builds: start/finish, result, duration, params         | Read jobs/builds (API token)                         | Запуск/остановка билдов, конфигурация jobs, admin             |


Принципы для сервисных аккаунтов:

- **Read-only** везде, где система это поддерживает. Monitor ничего не пишет обратно в Jira/GitLab/Jenkins.
- **Отдельный аккаунт на систему** — упрощает ротацию и отзыв, ограничивает радиус поражения при компрометации.
- **Ротация токенов** — владелец ротации фиксируется в [discovery.md](./discovery.md) §9.5 (`access_owners`).
- Scope токена сверяется при выдаче smoke-тестом (`GET /rest/api/2/myself`, `GET /api/v4/user`, `GET /api/json`) — см. [discovery.md](./discovery.md) §7.

## 5. Secrets

- **Только environment variables.** Все секреты (`JIRA_TOKEN`, `DELIVERY_MONITOR_ADMIN_TOKEN`, GitLab/Jenkins токены, `DB_PASSWORD`, …) передаются через env / секрет-хранилище стенда, а не через файлы в репозитории.
- **Разделение границ доверия — входящий admin-токен ≠ `JIRA_TOKEN`** ([ADR-012](./adr/0012-minimal-auth-baseline-admin-endpoints.md)). `JIRA_TOKEN` защищает **исходящее** направление (Monitor → Jira, Monitor как клиент); `DELIVERY_MONITOR_ADMIN_TOKEN` защищает **входящее** направление (клиент → admin API, Monitor как сервер). Это разные секреты: утечка одного **не** должна давать доступ ко второму направлению. Переиспользование одного токена для обоих направлений запрещено.
- **Никаких секретов в Git.** В `application.yml` секреты по умолчанию **пустые** и подставляются из env ([session_log.md](./session_log.md), Phase 2.1); `.env` с реальными значениями — в `.gitignore`. Запрет коммита секретов зафиксирован в [ai_context.md](./ai_context.md) §7.
- **Никаких секретов в логах** — см. §8.
- Fail-fast: отсутствие обязательного секрета (напр. `JIRA_TOKEN`) **не даёт приложению стартовать**, а не всплывает позже при первом вызове ([session_log.md](./session_log.md), gate-check Phase 2.1) — секрет не должен «тихо» подставляться заглушкой.
- Секреты не попадают в документацию `docs/` ([discovery.md](./discovery.md), Change policy).

## 6. Network

- **Внутренний сервис.** Monitor разворачивается во внутренней сети и **не публикуется** в интернет.
- **VPN.** Доступ пользователей и администраторов — из корпоративной сети / через VPN.
- **HTTPS.** Весь трафик (UI и API, а также исходящие вызовы к `https://jira.eltc.ru`, `https://git.eltc.ru`, Jenkins) — по TLS. HTTP без шифрования не используется.
- **Webhooks** (опциональные, от GitLab/Jenkins/Jira) принимаются только с whitelisted-источников внутренней сети; hostname Monitor согласуется с DevOps/сетевиками ([discovery.md](./discovery.md) §7, §9.5).
- Расположение за корпоративным периметром — основной сетевой контроль; Monitor не рассчитан на прямую экспозицию наружу.

## 7. Audit

Логируются (для расследования и подотчётности) как минимум следующие действия:


| Событие                                                                       | Что фиксируется                                      |
| ----------------------------------------------------------------------------- | ---------------------------------------------------- |
| Привилегированные операции (напр. `POST /api/admin/sync/jira`)                | Кто, когда, результат (`{ fetched, saved, errors }`) |
| Изменения конфигурации / ролей                                                | Кто, что изменил, когда                              |
| Ошибки аутентификации / отказ в доступе                                       | Субъект (без секретов), endpoint, время              |
| Запуск / остановка синхронизаций (scheduler, ручной sync)                     | Источник (Jira/GitLab/Jenkins), время, итог          |
| Обращения сервисных аккаунтов к внешним системам на уровне «выполнено/ошибка» | Система, статус, **без** тела токена                 |


Аудит-записи содержат актора, действие, время и результат — но **никогда** секреты или токены (§8).

> **До корпоративного SSO/OIDC** ([ADR-012](./adr/0012-minimal-auth-baseline-admin-endpoints.md)): admin-токен stateless и **не** несёт идентичности пользователя. Для привилегированных операций «актор» = обладатель admin-токена (+ источник/IP), а **не** именованный сотрудник. Персональная идентичность (principal) в аудите появляется только с OIDC — не приписывать действие конкретному человеку, пока идентичности нет. См. запись в [decisions.md](./decisions.md) (Design notes, 2026-07-15 — ADR-012 implementation note).

## 8. Logging

Запреты (жёсткие, применяются на всех уровнях логирования):

- **Запрещено логировать токены** — PAT, API-токены, `DB_PASSWORD`, любые значения секретов из §5.
- **Запрещено логировать** `Authorization` **header** (Basic и `Bearer`) и его содержимое — ни в debug, ни в trace, ни в дампах запросов/ответов к Jira/GitLab/Jenkins.
- Не логировать полные тела ответов, если они могут содержать чувствительные данные; при необходимости — усечённый/санитизированный фрагмент.
- Ошибки интеграций логируются как тип/статус (напр. `JiraClientException` + HTTP-статус), **без** заголовков авторизации ([session_log.md](./session_log.md), Phase 2.1).
- Уровни `DEBUG`/`TRACE` для HTTP-клиентов не должны включать дамп заголовков в средах, где логи доступны шире, чем секреты.

## 9. Security Checklist

Перед развёртыванием / при ревью изменений, влияющих на безопасность:

- [ ] Целевая аутентификация пользователей — корпоративный SSO/LDAP/OIDC; локальных пользователей нет (§2)
- [ ] До SSO: `/api/admin/**` защищён Bearer admin-токеном из env (baseline, [ADR-012](./adr/0012-minimal-auth-baseline-admin-endpoints.md)) (§2)
- [ ] Входящий `DELIVERY_MONITOR_ADMIN_TOKEN` — **отдельный** секрет, не равен `JIRA_TOKEN` (разные границы доверия) (§5)
- [ ] Роли Admin / PM / QA / Developer — целевая модель; в baseline реализован только admin-gate (§3)
- [ ] Привилегированные операции (`/api/admin/*`) доступны только с admin-токеном / роли Admin (§3)
- [ ] actuator, кроме `health`/liveness, не открыт публично (§2, [ADR-012](./adr/0012-minimal-auth-baseline-admin-endpoints.md))
- [ ] Read-эндпоинты (`GET /api/**`) — открыты только внутри VPN; пересмотреть при появлении UI/SSO (§6)
- [ ] Сервисные аккаунты Jira / GitLab / Jenkins — отдельные и **read-only** (§4)
- [ ] Scope токенов проверен smoke-тестом; избыточных прав нет (§4)
- [ ] Все секреты — из environment variables (§5)
- [ ] В Git нет секретов; `.env` в `.gitignore`; `application.yml` без реальных значений (§5)
- [ ] Отсутствие обязательного секрета → приложение не стартует (fail-fast) (§5)
- [ ] Monitor — внутренний сервис за VPN, наружу не опубликован (§6)
- [ ] Весь трафик по HTTPS/TLS (§6)
- [ ] Webhooks принимаются только с whitelisted-источников (§6)
- [ ] Привилегированные действия и отказы доступа попадают в audit-лог (§7)
- [ ] Токены и `Authorization` header **не** попадают в логи (§8)
- [ ] Владелец ротации токенов назначен ([discovery.md](./discovery.md) §9.5)
