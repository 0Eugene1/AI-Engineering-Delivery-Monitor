### Added

- **Phase 2.5 Scheduler implemented**: новый класс `ru.eltc.deliverymonitor.sync.jira.JiraSyncScheduler` — фоновый вход в тот же `JiraSyncService.syncBoard()`, который уже вызывает `POST /api/admin/sync/jira`; не вызывает `Controller`, не вызывает `JiraClient` напрямую, не обходит `JiraSyncService`. Реализован через `SchedulingConfigurer#configureTasks` (не аннотацию `@Scheduled`), чтобы регистрировать задачу условно (`jira.sync.enabled`, default `false`) с env-driven `Duration`-интервалом (`jira.sync.interval`, default `5m`); использует `ScheduledTaskRegistrar.addFixedDelayTask` (**не** `addFixedRateTask`) — следующий прогон планируется только после завершения предыдущего. `runScheduledSync()` логирует агрегаты результата и **не** пробрасывает исключения — сбой одного прогона не мешает следующему тику. `@EnableScheduling` добавлена на `DeliveryMonitorApplication` (включает механизм; активацию самой задачи решает `jira.sync.enabled`). `JiraSyncProperties` дополнен полями `enabled`/`interval`; `application.yml` — `jira.sync.enabled: ${JIRA_SYNC_ENABLED:false}`, `jira.sync.interval: ${JIRA_SYNC_INTERVAL:5m}`. **In-process guard** добавлен в `JiraSyncService` (единственное изменение существующего класса): `AtomicBoolean syncInProgress` — если manual и scheduled sync пересекаются по времени, второй вызов `syncBoard()` немедленно пропускается и возвращает существующий `JiraSyncResult` с сообщением в уже существующем поле `errors()` (`"Sync already in progress; this run was skipped"`) — **без** HTTP `409`, **без** изменения формы/контракта `JiraSyncResult`, без нового поля. Не добавлялись (по решению задачи): `sync_state` таблица, distributed lock (ShedLock/Redis), incremental sync, retry framework, новый persistence-слой. Тесты (новые, 5): `JiraSyncSchedulerTest` (4, unit, `JiraSyncService` — Mockito mock: disabled → нет регистрации задачи; enabled → `addFixedDelayTask` с верным интервалом, `addFixedRateTask` не вызывается; scheduled run вызывает `syncBoard()`; ошибка `syncBoard()` не пробрасывается наружу), `JiraSyncServiceTest` (+1 — реальная многопоточность через `Thread`+`CountDownLatch`: конкурентный вызов `syncBoard()` во время выполнения другого прогона пропускается немедленно без обращения к персистентности, исходный прогон не прерывается, флаг корректно снимается после завершения). `.\mvnw.cmd clean verify` — 75 тестов (было 70), 0 failures, 0 errors, 2 skipped.
