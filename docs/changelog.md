## [Unreleased]

### Added

- **Phase 2.5 Scheduler implemented**: новый класс `ru.eltc.deliverymonitor.sync.jira.JiraSyncScheduler` — фоновый вход в тот же `JiraSyncService.syncBoard()`, который уже вызывает `POST /api/admin/sync/jira`; не вызывает `Controller`, не вызывает `JiraClient` напрямую, не обходит `JiraSyncService`. Реализован через `SchedulingConfigurer#configureTasks` (не аннотацию `@Scheduled`), чтобы регистрировать задачу условно (`jira.sync.enabled`, default `false`) с env-driven `Duration`-интервалом (`jira.sync.interval`, default `5m`); использует `ScheduledTaskRegistrar.addFixedDelayTask` (**не** `addFixedRateTask`) — следующий прогон планируется только после завершения предыдущего. `runScheduledSync()` логирует агрегаты результата и **не** пробрасывает исключения — сбой одного прогона не мешает следующему тику. `@EnableScheduling` добавлена на `DeliveryMonitorApplication` (включает механизм; активацию самой задачи решает `jira.sync.enabled`). `JiraSyncProperties` дополнен полями `enabled`/`interval`; `application.yml` — `jira.sync.enabled: ${JIRA_SYNC_ENABLED:false}`, `jira.sync.interval: ${JIRA_SYNC_INTERVAL:5m}`. **In-process guard** добавлен в `JiraSyncService` (единственное изменение существующего класса): `AtomicBoolean syncInProgress` — если manual и scheduled sync пересекаются по времени, второй вызов `syncBoard()` немедленно пропускается и возвращает существующий `JiraSyncResult` с сообщением в уже существующем поле `errors()` (`"Sync already in progress; this run was skipped"`) — **без** HTTP `409`, **без** изменения формы/контракта `JiraSyncResult`, без нового поля. Не добавлялись (по решению задачи): `sync_state` таблица, distributed lock (ShedLock/Redis), incremental sync, retry framework, новый persistence-слой. Тесты (новые, 5): `JiraSyncSchedulerTest` (4, unit, `JiraSyncService` — Mockito mock: disabled → нет регистрации задачи; enabled → `addFixedDelayTask` с верным интервалом, `addFixedRateTask` не вызывается; scheduled run вызывает `syncBoard()`; ошибка `syncBoard()` не пробрасывается наружу), `JiraSyncServiceTest` (+1 — реальная многопоточность через `Thread`+`CountDownLatch`: конкурентный вызов `syncBoard()` во время выполнения другого прогона пропускается немедленно без обращения к персистентности, исходный прогон не прерывается, флаг корректно снимается после завершения). `.\mvnw.cmd clean verify` — 75 тестов (было 70), 0 failures, 0 errors, 2 skipped.
- **Read API implemented**: новый пакет `ru.eltc.deliverymonitor.api.issue` — `IssueController` (`GET /api/issues` — все persisted issues; `GET /api/issues/{key}` — одна issue по публичному Jira `key`, `404` с `{ "error", "code": "ISSUE_NOT_FOUND" }` если не найдена), тонкий HTTP-адаптер без бизнес-логики. `IssueQueryService` (`@Transactional(readOnly = true)`) — read-only слой, грузит `IssueEntity` через `IssueRepository` и маппит в новый DTO `IssueResponse` (`issueKey, summary, status, statusCategory, assigneeUsername, assigneeDisplayName, issueType, jiraCreated, jiraUpdated, fixVersions, labels`) **внутри** транзакции — иначе LAZY `@ElementCollection` (`fixVersions`/`labels`) кидали бы `LazyInitializationException` вне сессии. `IssueRepository` расширен методом `findByKey(String)`. Зависимость строго `PostgreSQL → domain.issue → api.issue`: `api.issue` не импортирует `sync.jira`/`integration.jira`/`JiraClient` — никаких live-запросов в Jira из read API. `/api/issues/**` остаётся `permitAll()` в существующем `SecurityConfig` — **не менялись**: `JiraSyncService`, `JiraClient`, `api.admin.JiraSyncController`, `api.security` (security baseline), существующий sync flow, Liquibase-схема (новых миграций нет). **Сознательно не реализован** `GET /api/sprints/current` — по прямому решению архитектурного review перед этим этапом: нет `sprints` persistence (`docs/database.md` — Planned/future); никакого mock/stub/live-Jira substitute вместо него не добавлено — TODO зафиксирован в `docs/discovery.md`. Никаких pagination/sorting/filtering/search. Тесты (новые, 7): `IssueControllerTest` (3, `@WebMvcTest` + реальный `SecurityConfig`, mock `IssueQueryService` — 200/200/404), `IssueQueryServiceIntegrationTest` (3, `@DataJpaTest` против настоящего Liquibase-changeset, реальная LAZY-загрузка коллекций), `DeliveryMonitorApplicationTests` (+1 — публичный доступ к `/api/issues` без токена). `.\mvnw.cmd clean verify` — 70 тестов (было 63), 0 failures, 0 errors, 2 skipped.
- **Admin Sync HTTP API (Phase 2.4, [roadmap.md](./roadmap.md), [ADR-012](./adr/0012-minimal-auth-baseline-admin-endpoints.md)) implemented**: первый REST controller и первый security-слой проекта. Новый пакет `ru.eltc.deliverymonitor.api.admin` — `JiraSyncController` (`POST /api/admin/sync/jira`), тонкий HTTP-адаптер над `sync.jira.JiraSyncService`, без бизнес-логики; реюзает существующий `JiraSyncResult` как response, без отдельного DTO. Новый пакет `ru.eltc.deliverymonitor.api.security` — минимальный enforcement-слой: `SecurityConfig` (`SecurityFilterChain` — `/actuator/health` открыт, `/api/admin/**` и любой другой `/actuator/**` (включая `/actuator/info`) требуют аутентификации, CSRF отключён, sessions `STATELESS`, отказ → `401` через `HttpStatusEntryPoint`, без redirect на login), `AdminTokenAuthenticationFilter` (`OncePerRequestFilter`, сравнивает `Authorization: Bearer <token>` с конфигом константным по времени сравнением, никогда не логирует токен, пустой сконфигурированный токен никогда не аутентифицирует), `AdminTokenProperties` (`delivery-monitor.admin.token` ⇐ `DELIVERY_MONITOR_ADMIN_TOKEN`, fail-fast как `JIRA_TOKEN`). Аутентификация **stateless**: подтверждает только «токен валиден», identity пользователя не извлекается — нет `User`/`Role` entity, `UserRepository`, principal model, permissions. Не добавлялись: JWT, OAuth2 Resource Server, OIDC, LDAP, UI, scheduler, webhook security, audit database, incremental sync, async execution, retry. `backend/pom.xml`: добавлены `spring-boot-starter-security` и `spring-security-test`. Тесты (новые, 14): `AdminTokenAuthenticationFilterTest` (6, unit), `AdminTokenPropertiesTest` (3, fail-fast), `JiraSyncControllerTest` (3, `@WebMvcTest` + реальный `SecurityConfig`, mock `JiraSyncService` — 200/401/401), `DeliveryMonitorApplicationTests` (+2, полный контекст — 401 на admin endpoint и на `/actuator/info` без токена). `.\mvnw.cmd clean verify` — 63 теста (было 49), 0 failures, 0 errors, 2 skipped. **Отклонение от эскиза `api.md`**: response не содержит поле `saved` (derived-метод `JiraSyncResult`, не record-компонент — Jackson не сериализует; sketch исправлен под фактический контракт). **Уточнение endpoint policy**: actuator-правило реализовано буквально — `/actuator/**` кроме `/actuator/health` явно `authenticated()`, а не оставлено на «остальное как было», иначе `/actuator/info` был бы публичным по умолчанию.
